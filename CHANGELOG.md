# 更新日志

本文件记录 skill-hub 项目的所有显著变更。

格式参考 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.1.0/)，版本号遵循 [语义化版本](https://semver.org/lang/zh-CN/)。

## [2.3.0] - 2026-07-07

### 破坏性变更
- SKILL.md 版本号从 2.2.0 升至 2.3.0
- **脚本执行不可降级**（issue-006/009）：校验脚本执行失败时禁止降级为 AI 手动校验，必须修复后重试
- **B1 质量校验取消降级路径**：原"脚本执行失败降级为 AI 手动校验"的路径被移除
- 索引分区从五分区扩展为**六分区**（新增架构专题）
- A2 业务动词聚类升级为**三维聚类**（动词+业务名词+修饰词变体）

### 新增
- 新增**脚本执行门控机制**（issue-006 核心修复）：6 个检查点（含门控0环境自愈），脚本未执行时流程不可继续
  - 门控0：环境前置检查（python3 缺失时自动安装，不阻塞流程）
  - 门控1：A1/A2 脚本执行（步骤3 后）
  - 门控2：A6 大类枚举（步骤3 后）
  - 门控3：A7 跨领域识别（步骤3 后）
  - 门控4：B1 质量校验（步骤6 每个文件）
  - 门控5：B4 代码反向验证（步骤6 每个文件）
- 新增**环境前置检查与自愈**：ensure-python3.sh 自动检测并安装 python3
  - 支持 macOS（brew）/ Ubuntu/Debian（apt）/ CentOS/RHEL（yum/dnf）/ Alpine（apk）
  - 同时校验基础命令（grep/awk/sed/find/wc）和脚本完整性
- 新增**A6 大类强制方法枚举**（issue-001）：扫描行数>500或方法数>15的类，列出所有方法
  - 新增脚本：scripts/extract-large-class-methods.sh
  - 大类阅读不受"每文件≤80行"限制，必须分批完整阅读
- 新增**A7 跨领域子场景识别**（issue-002）：识别跨模块调用链和高频状态枚举引用
  - 新增脚本：scripts/cross-domain-scanner.sh
  - AI 必须输出跨领域子场景登记表
  - 知识库索引增加"子场景路由表"
- 新增**A8 业务变体识别**（issue-005）：基于 A2 维度2+维度3 识别业务变体
- 新增**B4 代码反向验证**（issue-009）：枚举值/方法名/类名/路径在代码中反向验证
  - 新增脚本：validators/cross-validate-with-code.sh
  - 消除 AI 自校验的确认偏差
- 新增**subagent 深度阅读量化指令模板**（issue-003）：标准化 subagent 阅读要求
  - 全方法枚举、业务变体识别、调用链追踪3层、方法重载识别、状态枚举扫描
  - 输出覆盖率自检报告（methodCoverage/unreadMethods/identifiedVariants）
- 新增**进度状态文件** progress.json（issue-008）：机器可读的跨会话恢复状态
  - 记录 completedDomains/scannedClasses/nextBatch
  - 跨会话恢复不依赖用户记忆
- 新增**架构专题类模板**（issue-007）：横切架构特征的系统性描述
  - 新增模板：templates/架构专题类模板.md
  - 包含架构决策记录、架构拓扑、部署差异等章节
- 新增**领域标记"已完成"前置条件**（issue-004）：5 项前置检查
- 新增**三阶段消费+回退策略**（issue-010）：知识库未命中时的完整处理路径
  - 阶段1：索引查询（≤5秒）
  - 阶段2：命中质量评估（充分/部分/不足/未命中四级判定）
  - 阶段3：回退执行（记录到 missed-queries.log + 直接 Grep）
- 新增**知识库覆盖反馈闭环**（issue-010）：未覆盖场景自动记录 + 3次以上触发增量补充建议
- 业务领域类模板新增**架构上下文章节**（issue-007）：部署架构/设计模式/数据架构/外部依赖
- 项目导航类模板新增**架构特征清单**和**子场景路由表**章节（issue-007/002）
- 机器可读摘要模板新增 **coverageReport 字段**（issue-003/004）：覆盖率自检报告
- coverageGaps 新增 **gapType 字段**（core/detail）区分核心子场景和实现细节

### 变更
- **明确声明适用语言为 Java / JVM 系**：SKILL.md 顶部和模板表增加适用范围声明，清理多语言映射说明（避免误导非 JVM 项目用户）
- **A2 升级为三维聚类**（issue-005）：从单一动词聚类升级为动词+业务名词+修饰词三维交叉
  - 维度2：业务名词族（Red/Invalid/Invoice/Sign/Tax 等）
  - 维度3：修饰词变体标注（Part/Fast/Direct/ISV/Pre 等）
  - 输出业务变体矩阵
- **A4 覆盖深度校验升级**为"覆盖率+密度"双维度（issue-004）：
  - 方法覆盖率 ≥ 70%（新增）
  - 变体覆盖率 100%（新增）
  - 状态覆盖率 ≥ 90%（新增）
- **B1 质量校验不可降级**（issue-006/009）：禁止用 AI 手动 Read 替代脚本执行
- validate-knowledge.sh 新增**覆盖深度双维度校验**（校验6）：密度+覆盖率+待补充分类检查
- 业务领域类模板**待补充章节分类**（issue-004）：必须区分"核心子场景"（≤2条）和"实现细节"（无上限）
- 步骤3 增强识别从 A1-A5 扩展为**A1-A8**
- subagent 输出统一格式增加**覆盖率自检报告**
- 步骤5 分批策略优先级新增**架构专题**
- 模板版本号统一升至 2.3.0
- 配套工具表新增 3 个脚本/校验器的说明

### 修复
- 修复大类（God Class）中 40% 子场景被系统性遗漏的问题（issue-001）
- 修复跨领域子场景被"双重遗漏"（两个领域都不覆盖）的问题（issue-002）
- 修复 subagent 阅读深度不可控导致覆盖深度 55%-85% 不均衡的问题（issue-003）
- 修复"待补充"条目过多（27+条，含15条核心子场景）导致完成率虚高的问题（issue-004）
- 修复方法级业务变体识别缺失导致冲红5个变体无法关联的问题（issue-005）
- 修复配套脚本在生成过程中从未被执行（执行率0%）的问题（issue-006）
- 修复横切架构概念（云岛双部署等）分散在6个文档各描述20%的问题（issue-007）
- 修复跨会话恢复依赖用户记忆导致上下文丢失的问题（issue-008）
- 修复质量校验降级为AI手动校验导致确认偏差的问题（issue-009）
- 修复知识库未命中时缺少回退策略导致AI浪费时间无效查询的问题（issue-010）

## [2.2.0] - 2026-07-07

### 破坏性变更
- SKILL.md 版本号从 2.1.0 升至 2.2.0
- 核心理念转变：知识库首要服务对象从"人类"变为"AI"，目的是减少 AI 每次扫描范围
- 新增双产物机制：每个知识库文档必须同时生成 .md（人类可读）和 .summary.json（AI 可消费）

### 新增
- 新增**AI 可消费性设计**顶级章节（D1/D2/D3）
  - D1 知识库查询工具（query-knowledge.sh）：AI 执行任务前先查知识库，命中后消费 .summary.json 减少扫描
  - D2 机器可读摘要（.summary.json）：AI 查询的首要入口，含 entryPoints/coreChain/grepHints/confidence
  - D3 场景触发：用户理解/分析/设计/排查任务时先查知识库
- 新增**增强领域识别**（A1-A5），内聚在步骤3同一章节
  - A1 业务术语候选提取（extract-business-terms.sh）：候选字符串提取器，分词由 AI 完成
  - A2 业务动词聚类（cluster-by-business-verb.sh）：按动词分组方法名识别跨模块领域
  - A3 方法级重叠率判定：量化领域边界（≥50%合并，<30%独立）
  - A4 覆盖深度校验：检查已有文档的覆盖深度
  - A5 跨模块领域重组：3条聚合规则（术语/调用链/共享枚举）
- 新增**模式四：外部依赖补充**（C1）：合规反编译二方库
  - decompile-external-jar.sh：LICENSE 检查 + javap 结构提取
  - 合规优先：禁止反编译的 jar 跳过
  - 明确 javap 能力边界：只获取结构定义，业务含义需 AI 反推
  - 坐标定位修正：从 pom.xml 解析，不从类名反推
- 步骤8 新增**外部依赖清单**（C2）：索引底部维护外部库的 Maven 坐标和反编译状态
- 步骤6 新增**AI 可消费性校验**（B3）：检查 .summary.json 存在性和关键字段完整性
- 索引首次构建新增**外部依赖分区**
- 触发条件新增 D3 场景触发
- 新增模板：templates/机器可读摘要模板.json
- 新增脚本：scripts/query-knowledge.sh、scripts/extract-business-terms.sh、scripts/cluster-by-business-verb.sh、scripts/decompile-external-jar.sh

### 变更
- 步骤6 校验脚本**强制执行**（B1）：执行失败降级为 AI 手动校验，校验不通过强制修正
- validate-knowledge.sh **路径提取增强**（B2）：支持多种格式（Java 类名、反引号路径、表格裸路径、src/main 片段）
- validate-knowledge.sh 新增 **--json 参数**：输出结构化报告便于 AI 消费
- validate-knowledge.sh 新增**失败项明细**输出
- 评分卡新增 **AI 可消费性维度**（15分）和**领域复杂度弹性**（简单领域门槛降至65）
- description 更新：强调"为 AI 解决问题提供精准依据"
- 步骤3 A1-A5 五项增强成组执行，构成"增强领域识别"内聚章节
- 索引分区从四分区扩展为**五分区**（新增外部依赖）

## [2.1.0] - 2026-07-06

### 新增
- 新增**"并行执行与 subagent 协作规范"**章节，统一规范扫描阶段和生成阶段的 subagent 协作：
  - 扫描阶段：中型项目强制 subagent 并行，定义模块划分、并行规模、统一输出格式
  - 生成阶段：模式B 深度阅读委托 subagent，定义代码依据标注规范和主流程抽样验证机制
  - subagent 容错：空结果/格式异常时记录失败模块并重试，不阻塞其他模块
- 步骤2 新增**首次构建分支**：知识库目录或索引不存在时，一次性创建目录 + 四分区表头索引（项目导航/工具框架/业务领域/数据模型），给出索引初始结构示例
- 新增**数据模型类模板**（`templates/数据模型类模板.md`）：覆盖核心表、字段字典、枚举值、关系图，解决表结构类知识无处沉淀的问题
- 步骤5 新增**批次进度档案**：索引顶部维护"待生成领域清单"，支持跨会话恢复模式
- 步骤5 新增**深度阅读分流**：领域涉及类 ≤10 主流程直读（模式A），>10 委托 subagent（模式B）
- 步骤8 新增**关键词数量指引**：每文件 10-25 个，覆盖中英文，避免堆砌包名
- 步骤8 新增**关键词分隔符格式**：统一 ` / ` 分隔（前后含空格）
- 通用异常处理表新增 **subagent 返回空/格式异常**的处理策略
- 触发条件新增**恢复模式**触发：用户说"继续生成知识库"/"继续扫描"时自动恢复

### 变更
- 步骤5 **优先级排序细化**：项目导航 > 跨模块核心链路 > 单模块业务领域 > 数据模型 > 工具框架 > 外部集成

- 步骤7 **协作约束文件更新条件放宽**：首次构建时强制追加知识库入口引用，后续保持保守策略
- 步骤5 新增**命名规则示例**：通用概念用中文，专有名词保留英文
- 步骤8 关键词提取规则新增**数据模型类**的关键词类型
- 索引初始结构新增**数据模型类分区**
- 业务领域类模板新增**状态流转独立小节**（含 ASCII 状态机示例）
- 业务领域类模板**高风险点格式细化**：要求类型标签（`[可靠性]`/`[并发]`/`[代码缺陷]`/`[安全]`/`[性能]`）+ 位置（文件+行号）+ 影响场景 + 建议处理

- 上下文保护规则新增第7条：中型项目强制 subagent
- 三套已有模板版本号升至 2.1.0

## [2.0.0] - 2026-07-06

### 破坏性变更
- 联动维护规则写入策略从"主动写入"改为"草稿确认制"：首次生成知识库后，不再未经确认直接写入协作约束文件，而是生成草稿内容，由用户确认后再写入
- SKILL.md 版本号从 1.1.0 升至 2.0.0

### 新增
- 新增**模式三：知识库体检（失效扫描）**，反向扫描知识库引用的类名/方法名/路径在代码中是否存在，输出失效报告
- 新增**通用异常处理**章节，覆盖索引格式异常、超大文件、二进制文件、代码矛盾、工具降级 5 类异常场景
- 新增**协作约束文件降级策略**：项目中无协作约束文件时，自动在根目录创建通用 `AGENTS.md`（基于 `templates/AGENTS.md` 模板）
- 新增**上下文注入检查**：步骤1 检查协作约束文件是否已注入 AI 上下文，未注入时告知用户配置
- 步骤6 新增**核心链路回溯校验**：对每条调用链反向在代码中验证方法名存在性、调用关系成立性、状态值有效性
- 新增 `scripts/` 目录，提供 Java/Python/JS 三种语言的类签名提取脚本
- 新增 `validators/` 目录，提供质量校验脚本（`validate-knowledge.sh`）和失效扫描脚本（`audit-knowledge.sh`）
- 新增 `templates/AGENTS.md` 通用协作约束文件模板
- 步骤2 新增**覆盖判定规则表**，明确类名/概念/表名/方法名/模块的覆盖判定方式
- 模式二步骤4 新增**「补充 vs 新建」判定标准表**

### 变更
- **触发条件**重构为可操作规则（动作词 + 沉淀意图词组合判定），消除"暗示""明确指向"等主观判断词
- 步骤3 第二层扫描新增**批次量化标准**（每批 ≤ 20 文件、每文件 ≤ 80 行），并明确脚本优先、人工降级的执行方式
- 步骤6 和步骤8 新增**去重闭环**：校验发现关键词重复时必须自动去重并重新校验，而非仅输出建议
- 三套模板新增**版本标识**、**空值处理规则**和（业务领域类）**多语言映射表**
- 业务领域类模板补充 Python/Go/Node.js 的字段概念映射说明
- 多语言适配说明：明确声明当前版本充分适配 Java/JVM 系，其他语言通过字段映射使用
- `description` 更新，体现知识库体检能力

### 修复
- 修复触发条件与不触发场景依赖主观判断词导致边界模糊的问题
- 修复质量校验缺少"代码事实一致性"校验的问题（新增核心链路回溯校验）
- 修复协作约束文件找不到时无降级路径导致后续步骤失去输入的问题
- 修复联动维护规则"主动写入"的越权风险（改为草稿确认制）
- 修复覆盖判定无容差定义导致误判的问题（明确"完全匹配才算覆盖"）
