# 知识库生成器（knowledge-generator）

本 skill 用于从项目代码中自动识别未被知识库覆盖的核心业务逻辑，生成知识库文件，并同步更新知识库索引。

## 触发条件

- 用户明确要求"生成知识库"/"补充知识库"/"沉淀知识"/"写知识库文档"/"维护知识库"等
- 用户明确给出想沉淀为知识库的逻辑方向（如"退款链路"、"迁移逻辑"等）
- 用户明确要求"扫描项目生成知识库"/"自动生成知识库"/"扫描知识库缺口"
- 用户要求"梳理业务逻辑"/"整理代码逻辑"/"写个文档总结一下"/"整理代码知识"且上下文暗示需要沉淀为知识库文档时
- 代码修改完成后，涉及知识库已覆盖的模块、类、业务流程，根据项目协作约束文件中的维护规则应主动更新知识库时

**不触发场景**：
- 用户只是咨询某个业务逻辑、排查问题、要求改代码或写设计文档，但没有明确要求沉淀知识库
- 用户只是提到"知识库"作为背景信息，而非要求生成、补充或维护知识库
- 用户的"梳理"/"整理"诉求明确指向设计文档而非知识库文档时

---

## 工作流程

### 模式一：自动扫描

当用户要求自动扫描项目生成知识库时，按以下步骤执行：

#### 步骤1：项目协作约束文件与知识库入口前置检查

在扫描代码之前，先检查项目中的协作约束文件和知识库入口，判断是否已存在知识库相关定义，以及是否有可沉淀到知识库中的内容。

**项目协作约束文件**：即项目在与 AI 互动时始终包含在对话上下文中的规约文件。Trae 中通过 `.trae/rules/` 目录或项目根目录约定文件（如项目协作约定文件、README 等）实现。不同项目的约束文件名称和位置可能不同，应通过检查项目根目录和 `.trae/rules/` 目录来确定。

**默认扫描哪些文件**：
- 项目协作约束文件（自动检测项目根目录和 `.trae/rules/` 目录）
- `docs/knowledge/知识库索引.md`（或其他项目约定的知识库索引入口）

**按需扫描哪些文件**：
- `docs/knowledge/` 下被索引命中的详细文档
- 历史需求文档和历史设计文档仅当用户明确指定并要求参考时才读取，且只能作为辅助背景理解，**绝对不能作为知识库内容的权威依据**
- 知识库内容的权威依据只能是当前代码事实；如果历史文档描述与当前代码不一致，以当前代码为准，并在输出中说明差异

**提取什么内容**：
- 项目模块结构、模块职责定义
- 所有引用知识库目录的路径
- 知识库索引的引用和格式
- 业务规则描述和技术框架使用说明
- 排查路径和定位规则
- 命名规范、注释规范等工具性内容

**输出**：
- 项目模块结构清单（模块名 + 职责描述 + 源码目录）
- 已存在哪些知识库定义（路径列表）
- 哪些规范内容已被知识库覆盖
- 哪些规范内容未被知识库覆盖（可作为知识库生成候选）

#### 步骤2：扫描现有知识库覆盖范围

- 读取知识库索引文件（默认 `docs/knowledge/知识库索引.md`），提取所有已覆盖的关键词、文件路径和涉及模块
- 不默认读取所有已有知识库文件；先基于索引表判断覆盖范围，再使用当前环境可用的文本检索工具按关键词、类名、表名检索知识库目录
- 只读取检索命中的少量详细文档，提取其中已引用的核心类名、业务概念、表名、枚举和调用链
- 如果索引缺失或格式异常，先明确指出索引问题，再按最小读取原则继续

#### 步骤3：分层渐进扫描识别未覆盖逻辑

**为避免一次性全量读取所有类文件导致上下文撑爆，必须严格遵循三层渐进扫描策略**。

**第一层：目录结构扫描（轻量级）**

只扫描包名和类名，不读取类文件内容。使用当前环境可用的文件枚举工具获取各模块下的目录结构和源码文件列表。

扫描范围：根据步骤1获取的项目模块结构清单来确定各模块的源码目录，不要假设模块结构是固定的。常见项目结构参考：
- Java 项目：业务层模块通常包含 service、manager、facade、core、workflow、event、mq、extPt 等子包；Web 层包含 controller、handler；数据层包含 dao、repository、mapper；组件层包含 job、failover、converter、lock 等
- 其他语言/框架：根据项目实际结构确定，不假设固定模式

**注意**：以上仅为常见项目结构参考。不同项目、不同语言的模块结构完全不同，必须以步骤1获取的实际项目结构为准。

输出：各模块下的类名清单，按包归类。

**第二层：摘要式扫描（中等消耗）**

对第一层扫描得到的类列表，只读取每个类的头部注释（类级文档注释）和 public 方法签名（方法名+参数列表），不读取方法体实现。

建议执行方式：
- 优先使用结构化解析、IDE/语言服务能力或脚本提取类级文档与 public 方法签名
- 若只能读取文件内容，应按小批次读取，并只截取必要片段
- 禁止在识别阶段批量打开完整源码文件并阅读全部方法体

此阶段目的是：
- 判断类是否有业务含义描述
- 判断类名是否已在知识库引用中
- 判断类的职责属于哪个业务领域
- 判断是否有未被知识库覆盖的核心逻辑

识别标准：
- 类注释含业务含义描述，且类名、业务概念、核心表名、核心枚举均未在知识库索引或命中文档中覆盖
- 类是核心业务入口（Controller/Facade/Manager/ServiceImpl 等），且入口方法、关键状态流转未被现有知识库覆盖
- 类的职责与某个领域强相关，但该领域目录、索引关键词、关联表名或关键枚举没有对应知识库文件
- 方法签名中包含明显的业务操作关键词，且现有知识库未覆盖该子场景
- 同一业务领域已有知识库时，优先判断是否补充现有文档，而不是新建重复文档

输出：按业务领域归类的未覆盖逻辑清单（领域名称 + 涉及类名列表 + 类职责简述）。

**第三层：深度代码阅读（仅在用户选定领域内执行）**

用户确认要生成哪些领域的知识库后，只对选定领域涉及的类做深度阅读（读取方法体、调用链、关联 DAO/Mapper 等）。此阶段消耗的上下文可控，因为只聚焦在1-2个领域范围内。

#### 步骤4：向用户展示扫描结果并确认

- 列出识别到的未覆盖领域清单（领域名称 + 涉及类数 + 职责简述）
- 向用户说明：建议每次只选择1-2个领域生成知识库，避免上下文过长影响输出质量
- 让用户选择哪些需要生成知识库

#### 步骤5：分批生成知识库文件

**为避免上下文撑爆，必须严格遵循分批生成策略**：
- 每次最多生成1-2个知识库文件
- 如果用户选择了3个以上领域，按优先级排序（核心业务编排类数量 > 服务实现类数量 > 其他），告知用户当前批次生成哪些，剩余的将在后续批次中处理
- 每个知识库文件生成前，先重新阅读该领域涉及的类文件（深度阅读），确保知识库内容准确

文件放置规则：
- 项目导航类知识 → `docs/knowledge/` 根目录下（文件名格式：`{中文主题}.md`）
- 工具/框架类知识 → `docs/knowledge/项目工具/` 下
- 业务领域类知识 → `docs/knowledge/业务领域/{领域名称}/` 下

文件命名规则：
- 文件夹名和文件名，除工具名、组件名、框架名等特定英文名词外，固定使用中文

#### 步骤6：质量校验

生成知识库文件后，必须执行以下自检：

1. **路径校验**：检查核心文件表中列出的所有文件路径是否在项目中真实存在，不存在则标注并修正
2. **关键词重复校验**：检查新增关键词是否与知识库索引中已有的关键词重复，避免同一关键词映射到多个知识库文件
3. **核心类完整性校验**：检查该业务领域的核心入口类是否全部覆盖，遗漏则补充
4. **交叉引用校验**：检查知识库文件中引用的其他知识库文档路径是否正确，引用的枚举、状态、表名是否与代码一致
5. **索引格式校验**：检查追加到知识库索引的行是否保持了原有表的列名、列数和格式

校验发现的问题应在输出中明确列出，并给出修正建议。

#### 步骤7：更新项目协作约束文件引用路径

- 默认不更新项目协作约束文件，只更新知识库文件和知识库索引
- 只有新增或调整长期稳定的协作规则、项目结构、模块职责、目录位置、外部依赖、业务流程或领域知识引用规则时，才更新对应协作约束文件或知识库下对应文档
- 如果确需更新协作约束文件，必须先读取目标文件了解章节结构，再决定在哪个章节追加引用；不要假设文件结构是固定的
- 只更新与新增知识库内容直接相关的章节，不做无关修改

#### 步骤8：更新知识库索引

在对应区块追加关键词映射行：
- 项目导航类 → 索引中对应的项目导航类区块
- 工具/框架类 → 索引中对应的工具/框架类区块
- 业务领域类 → 索引中对应的业务领域类区块

追加行时必须保持目标表的原有列名、列数和格式，不强行改表头。不同项目的索引区块名称可能不同，应以实际索引文件的区块名为准。

关键词提取规则（按知识库类型区分）：
- 项目导航类：概念名、模块名、子模块名，多个关键词用 `/` 分隔
- 工具/框架类：框架名、工具名、配置键名、类名、接口名
- 业务领域类：类名、接口名、方法名（驼峰和下划线形式）、核心枚举名、业务概念中文名、数据表名、模块包名

### 模式二：用户指定逻辑方向

当用户给出具体的逻辑方向时，按以下步骤执行：

#### 步骤1：项目协作约束文件前置检查

同自动扫描的步骤1，先检查协作约束文件和知识库索引中是否已有与用户指定逻辑方向相关的知识库定义或可沉淀内容。如果已有相关定义，直接引用并告知用户，避免重复生成。

#### 步骤2：解析用户意图

- 如果用户输入模糊（如只说"退款"），**必须主动提问**，不能自行猜测：
  - "你说的退款是指退款账单创建、退款审核还是冲正退款？"
  - "你想覆盖的深度是全链路还是某个特定子场景？"
- 提取核心关键词（类名、方法名、业务概念、模块名）

#### 步骤3：精准定位代码

- 根据关键词在代码中搜索相关类、方法、接口
- 读取核心文件，理解调用链和业务逻辑
- 遵循最小读取原则，只读取与目标逻辑直接相关的文件

#### 步骤4：检查已有覆盖

- 在知识库索引中检查是否已有相关知识库文件
- 如果已有，评估是否需要补充/扩展还是新建，向用户确认

#### 步骤5：生成知识库文件

同自动扫描的步骤5（分批生成策略同样适用）。

#### 步骤6：质量校验

同自动扫描的步骤6。

#### 步骤7：更新引用和索引

同自动扫描的步骤7和8。默认只更新知识库索引；项目协作约束文件仅在符合长期稳定规则变更条件时更新。

## 项目协作约束文件更新规则

项目协作约束文件不是业务知识的默认承载位置。根据知识库文件的类型，只有在确需同步长期稳定规则或入口引用时，才决定在协作约束文件的哪个章节添加引用：

| 知识库类型 | 协作约束文件章节参考 |
| ------ | ------ |
| 项目导航类 | 项目结构与问题定位章节 |
| 工具/框架类 | 对应功能章节（注释→注释章节，设计→设计章节，测试→测试章节，外部依赖→外部依赖章节） |
| 业务领域类 | 默认不追加到协作约束文件；只有影响长期稳定协作规则或知识库前置检查规则时才补充说明 |

**重要**：不同项目的协作约束文件结构可能不同，更新时应先读取文件了解其章节结构，再决定在哪个章节追加引用。不要假设文件结构是固定的。历史文档只能作为辅助参考，不能作为协作约束文件的事实来源；若历史文档与当前代码不一致，以当前代码为准。

## 联动维护规则

### 项目协作约束文件

每个项目在与 AI 互动时，都有一个始终包含在对话上下文中的项目协作约束文件（Trae 中通过 `.trae/rules/` 目录或项目根目录约定文件实现）。该文件定义了项目级别的协作行为规约。

### 知识库维护联动

本 skill 在生成知识库后，应在项目协作约束文件中确认以下维护规则已存在：

> 当任务涉及代码修改，且修改内容覆盖知识库已记录的模块、类、业务流程时，必须主动检查并同步更新对应知识库文档。

如果项目协作约束文件中尚未包含此规则，本 skill 应建议用户将其添加到项目协作约束文件中，以确保后续所有代码修改任务都能自动触发知识库维护检查。

**注意**：不同项目的协作约束文件名称和位置可能不同，不应硬编码具体文件名。应通过检查项目根目录和 `.trae/rules/` 目录来确定当前项目使用的协作约束文件。

## 模糊输入提问规则

当用户输入无法精准识别时，**必须主动提问**，按以下维度：

1. **业务领域确认**：你指的是哪个业务领域？（列出可能的选项）
2. **覆盖深度确认**：你想要全链路知识还是特定子场景？
3. **关键词确认**：你提到的{关键词}在项目中有多个对应类，你指的是哪个？（列出具体类名和路径）

**绝对禁止**：自行猜测用户意图并直接生成。必须先确认再执行。

## 上下文保护规则

1. **分层渐进扫描**：第一层只扫描目录结构→第二层只读类注释和方法签名→第三层只在选定领域内做深度阅读
2. **分批生成**：每次最多生成1-2个知识库文件，3个以上领域按优先级排序分批处理
3. **最小读取原则**：每个阶段只读取当前阶段所需的最少量文件
4. **禁止全量读取方法体**：在识别未覆盖逻辑阶段，只读类注释和方法签名，不读方法体
5. **历史文档非权威**：历史需求/设计文档仅作为辅助背景理解，知识库内容的权威依据只能是当前代码
