# 知识库生成器（knowledge-generator）

本 skill 用于从项目代码中自动识别未被知识库覆盖的核心业务逻辑，生成知识库文件，同步更新知识库索引，并能反向扫描知识库中已失效的引用。

---

## 触发条件

触发判定遵循**可操作规则**，不依赖"暗示""明确指向"等主观判断词。

**触发（满足以下任一条件即触发）**：

1. 用户诉求中同时出现「动作词」+「沉淀意图词」：
   - 动作词：生成 / 补充 / 沉淀 / 写 / 维护 / 整理 / 梳理 / 扫描
   - 沉淀意图词：知识库 / 知识文档 / 后续查阅 / 长期留存
   - 例："梳理退款链路沉淀到知识库" → 触发
2. 用户明确要求"扫描项目生成知识库"/"自动生成知识库"/"扫描知识库缺口"
3. 用户明确给出想沉淀为知识库的逻辑方向（如"退款链路"、"迁移逻辑"），且语句中含"知识库""沉淀""留存"之一
4. 代码修改完成后，根据联动维护规则需主动更新知识库（见"联动维护规则"章节）
5. 用户要求"体检知识库"/"扫描知识库失效引用"/"检查知识库是否过期"
6. 用户要求"继续生成知识库"/"继续扫描"，触发恢复模式（见步骤5"批次进度档案"）

**不触发（满足以下任一条件即不触发）**：

1. 用户只咨询业务逻辑、排查问题、要求改代码，**未出现任何沉淀意图词**
2. 用户明确要求"设计文档""方案文档""PRD""技术方案"，而非"知识库"
3. 用户只是提到"知识库"作为背景信息（如"参考下知识库里的退款逻辑"），而非要求生成/补充/维护

**模糊地带处理**：当无法判定是否触发时（如用户只说"帮我梳理下退款逻辑"，没有沉淀意图词也没有设计文档意图词），**默认反问澄清**："你是想沉淀到知识库，还是只是梳理逻辑/写设计文档？"，不自行猜测。

## 通用异常处理

以下异常在任意步骤出现时，按统一策略处理，避免执行中断：

| 异常场景 | 处理策略 |
| --- | --- |
| 索引文件格式异常且无法解析 | 终止当前步骤，输出"索引文件格式异常，无法解析"并列出异常片段，由用户决定是否手动修复后继续 |
| 目标文件超大（单文件 > 5000 行） | 跳过全量读取，只读取头部注释 + 方法签名，输出"文件过大已跳过方法体" |
| 二进制文件 / 非 UTF-8 编码 | 直接跳过，输出"已跳过非文本文件：{路径}" |
| 选定领域代码自相矛盾（如同名枚举值冲突） | 以当前代码中最后一次定义为准，在输出中明确标注冲突点 |
| 工具不可用（如 AST 提取脚本执行失败） | 降级为逐文件头部读取，输出"工具降级：改为逐文件读取" |
| subagent 返回空结果或格式异常 | 记录失败模块，在下一轮重试该模块，不阻塞其他模块的处理；重试仍失败则在输出中明确标注"模块 {X} 扫描失败，建议人工补充" |

## 工作流程

### 模式一：自动扫描

#### 步骤1：协作约束文件与知识库入口前置检查

在扫描代码之前，先检查项目协作约束文件和知识库入口。

**1.1 检测协作约束文件**

按以下优先级检测：

1. `.trae/rules/` 目录下的规则文件
2. 项目根目录的约定文件（`AGENTS.md`、`CLAUDE.md`、`.cursorrules`、`README.md` 等）

**降级策略（无协作约束文件时）**：

若以上检测均未找到协作约束文件，则在**项目根目录创建 `AGENTS.md`**，内容参照 [templates/AGENTS.md](templates/AGENTS.md) 模板。创建前必须告知用户："项目中未找到协作约束文件，将在根目录创建通用 `AGENTS.md`，后续 AI 会话将基于该文件协作。"创建完成后，将 `AGENTS.md` 作为协作约束文件继续后续步骤。

**1.2 检查协作约束文件是否已注入 AI 上下文**

联动维护规则的前提是协作约束文件被注入到每次 AI 会话上下文。检查方式：

- Trae 项目：检查 `.trae/rules/` 目录是否存在且包含规则文件（Trae 会自动注入该目录）
- 非 Trae 项目或 `AGENTS.md` 模式：在输出中明确告知用户："联动维护规则依赖协作约束文件被始终注入上下文。若你的 IDE 未将 `AGENTS.md`/约定文件配置为始终包含，请在 IDE 设置中配置，否则代码修改后不会自动触发知识库维护。"

**1.3 扫描协作约束文件和知识库索引**

**默认扫描**：
- 检测到的协作约束文件（含新建的 `AGENTS.md`）
- `docs/knowledge/知识库索引.md`（或其他项目约定的索引入口）

**按需扫描**：
- `docs/knowledge/` 下被索引命中的详细文档
- 历史需求/设计文档仅在用户明确要求时读取，**只能作为辅助背景，绝对不能作为知识库内容的权威依据**。若历史文档与当前代码不一致，以代码为准并说明差异

**提取内容**：
- 项目模块结构、模块职责定义
- 所有引用知识库目录的路径
- 知识库索引的引用和格式
- 业务规则描述和技术框架使用说明
- 排查路径和定位规则

**输出**：
- 项目模块结构清单（模块名 + 职责描述 + 源码目录）
- 协作约束文件路径（含是否新建的 `AGENTS.md`）
- 协作约束文件是否已注入上下文的判定结果
- 已存在的知识库定义（路径列表）
- 未被知识库覆盖的规范内容（生成候选）

---

#### 步骤2：扫描现有知识库覆盖范围

读取知识库索引文件，提取已覆盖的关键词、文件路径和涉及模块。

**首次构建分支（知识库目录或索引不存在时）**：

若 `docs/knowledge/` 目录或索引文件不存在，判定为**首次构建**。首次构建处理：

1. 创建 `docs/knowledge/` 目录
2. 创建索引文件 `docs/knowledge/知识库索引.md`，一次性补齐全部分区表头（项目导航类 / 工具框架类 / 业务领域类 / 数据模型类），暂无内容的分区标注"（待补充）"
3. **首次构建时，必须在协作约束文件中追加"知识库入口"引用**（见步骤7），确保后续任务能感知知识库存在
4. 后续步骤照常执行

索引文件初始结构示例：

```markdown
# 知识库索引

> 待生成领域清单（批次进度档案）：
> | 领域名称 | 涉及类数 | 状态 |
> | --- | --- | --- |
> | （扫描后填充） | | 待生成 |
>
> 维护规则：当代码修改覆盖知识库已记录的模块/类/流程时，必须同步更新对应知识库文档。

## 项目导航类

| 关键词 | 知识库文件 |
| --- | --- |
| （待补充） | |

## 工具框架类

| 关键词 | 知识库文件 |
| --- | --- |
| （待补充） | |

## 业务领域类

| 关键词 | 知识库文件 |
| --- | --- |
| （待补充） | |

## 数据模型类

| 关键词 | 知识库文件 |
| --- | --- |
| （待补充） | |
```

**覆盖判定规则**：

| 判定维度 | 判定方式 | 结论 |
| --- | --- | --- |
| 类名 | 完全匹配（区分大小写） | 已覆盖 |
| 业务概念 | 命中索引关键词的完全匹配或精确子串 | 已覆盖 |
| 数据表名 | 完全匹配 | 已覆盖 |
| 方法名 | 驼峰 / 下划线形式任一完全匹配 | 已覆盖 |
| 模块/包名 | 完全匹配或精确前缀匹配 | 已覆盖 |

**重要**：模糊语义相似**不算**覆盖。"账单"已覆盖不等于"账单明细"已覆盖；"退款"已覆盖不等于"冲正退款"已覆盖。宁可多报候选，交由步骤4人工过滤，也不漏报。

索引缺失或格式异常时，按"通用异常处理"策略执行。

---

#### 步骤3：分层渐进扫描识别未覆盖逻辑

**为避免上下文撑爆，必须严格遵循三层渐进扫描。中型项目（> 200 个类或 > 3 个模块）的扫描必须委托 subagent 并行执行，详见"并行执行与 subagent 协作规范"章节。**

**第一层：目录结构扫描（轻量级）**

只扫描包名和类名，不读取文件内容。使用文件枚举工具获取各模块的目录结构和源码文件列表。

扫描范围以步骤1获取的项目模块结构清单为准，不假设固定模式。常见结构参考：
- Java：service / manager / facade / controller / dao / job 等
- Python：app / services / models / views / tasks 等
- Go：cmd / internal / pkg / api 等
- Node.js：src / routes / services / models 等
- 其他语言：以实际目录结构为准

输出：各模块下的类名清单，按包/目录归类。

**第二层：摘要式扫描（中等消耗）**

对第一层得到的类列表，只读取**类级文档注释**和**公开成员签名**（方法名 + 参数列表 + 返回类型），不读方法体。

**执行方式（按优先级降级）**：

1. **优先**：使用配套脚本提取签名（见"配套工具"章节）：
   - Java：`scripts/extract-signature-java.sh`
   - Python：`scripts/extract-signature-python.py`
   - JS/TS：`scripts/extract-signature-js.mjs`
2. **降级**：脚本不可用时，按**每批 ≤ 20 个文件**分批读取，每个文件只读取前 80 行（覆盖类注释 + 方法签名），禁止整文件读取。

**批次量化**：单次读取不超过 20 个文件、每个文件不超过 80 行。大项目（> 200 个类）分多轮执行，每轮之间输出阶段性结果。

识别标准（结合步骤2的覆盖判定规则）：
- 类注释含业务含义描述，且类名 / 业务概念 / 核心表名 / 核心枚举均未被知识库覆盖
- 类是核心业务入口（Controller / Facade / Manager / ServiceImpl 等），且入口方法、关键状态流转未被覆盖
- 类的职责与某领域强相关，但该领域没有对应知识库文件
- 方法签名中包含明显的业务操作关键词，且知识库未覆盖该子场景
- 同一业务领域已有知识库时，优先判断是否补充现有文档，而不是新建重复文档

输出：按业务领域归类的未覆盖逻辑清单（领域名称 + 涉及类名列表 + 类职责简述）。

**第三层：深度代码阅读（仅在用户选定领域内执行）**

用户确认要生成哪些领域后，只对选定领域涉及的类做深度阅读（方法体、调用链、关联 DAO / Mapper 等）。

**业务领域划分原则（解决领域颗粒度无标准问题）**：

- 一个领域至少有 **3 个**强相关的核心类
- 一个领域对应一个**可独立排查的业务场景**（能回答"这块出问题怎么查"）
- 领域之间的核心类重叠率应 **< 30%**；超过则考虑合并
- 单领域知识库文档控制在 **300 行以内**，超出考虑拆分子领域

---

#### 步骤4：向用户展示扫描结果并确认

- 列出识别到的未覆盖领域清单（领域名称 + 涉及类数 + 职责简述）
- 告知用户：建议每次只选择 1-2 个领域生成，避免上下文过长影响输出质量
- 让用户选择哪些需要生成

---

#### 步骤5：分批生成知识库文件

**分批策略**：每次最多生成 1-2 个知识库文件。用户选择 3 个以上领域时，按以下优先级排序分批：

**优先级排序（细化）**：
1. 项目导航类（全局结构，优先级最高）
2. 跨模块核心业务链路（涉及 2 个以上模块的主流程）
3. 单模块业务领域
4. 数据模型类（表结构、字段字典、枚举值）
5. 工具/框架类
6. 外部集成类

每批生成前，告知用户当前批次和剩余批次。

**批次进度档案（解决批次衔接与跨会话恢复问题）**：

在知识库索引顶部维护"待生成领域清单"，记录每个领域的状态：

| 领域名称 | 涉及类数 | 状态 | 备注 |
| --- | --- | --- | --- |
| 发票开具主链路 | 32 | 待生成 | 跨模块核心链路 |
| 发票抬头管理 | 8 | 待生成 | 单模块领域 |

每批完成后更新状态（待生成 → 已生成 → 已校验）。

**恢复模式**：用户下次说"继续生成知识库"/"继续扫描"时，读取进度档案，自动从上次中断处继续，无需重新扫描。

**深度阅读分流（解决深度阅读与最小读取的冲突）**：

根据选定领域涉及的类数，选择深度阅读模式：

| 模式 | 触发条件 | 执行方式 |
| --- | --- | --- |
| 模式 A | 领域涉及类 ≤ 10 | 主流程内直接深度阅读，读取方法体和调用链 |
| 模式 B | 领域涉及类 > 10 | 委托 subagent 在独立上下文完成深度阅读 + 文档草稿生成，主流程只负责质量校验与落盘（见"并行执行与 subagent 协作规范"） |

**文件放置规则**：
- 项目导航类 → `docs/knowledge/` 根目录（文件名：`{中文主题}.md`）
- 工具/框架类 → `docs/knowledge/项目工具/`
- 业务领域类 → `docs/knowledge/业务领域/{领域名称}/`
- 数据模型类 → `docs/knowledge/数据模型/`

**命名规则**：文件夹名和文件名，除工具名、组件名、框架名等特定英文名词外，固定使用中文。

**命名规则示例**：
- 通用中文概念用中文：发票开具、发票抬头、税收分类
- 含专有名词保留英文：ISV开票、Nuonuo对接、OAuth鉴权

---

#### 步骤6：质量校验

生成知识库文件后，**必须执行全部校验**。校验分两层：第一层为程序化校验（优先使用 `validators/validate-knowledge.sh`），第二层为 AI 回溯校验。

**第一层：程序化校验**

| 序号 | 校验项 | 说明 |
| --- | --- | --- |
| 1 | 路径存在性 | 核心文件表中列出的所有文件路径在项目中真实存在 |
| 2 | 关键词重复 | 新增关键词不与索引已有关键词重复 |
| 3 | 索引格式 | 追加到索引的行保持原有列名、列数、格式 |
| 4 | 交叉引用 | 引用的其他知识库文档路径正确，引用的枚举、状态、表名存在 |
| 5 | 模板字段完整 | 生成的文件覆盖模板所有必填章节（缺项章节允许删除，但不得遗漏必填项） |

**第二层：核心链路回溯校验（AI 执行，解决"代码事实一致性"问题）**

对生成的知识库文件中每一条"入口 → 关键逻辑"调用链，反向在代码中验证：

1. 链路中出现的每个方法名，在代码中 grep 确认存在
2. 链路中"A → B"的调用关系，在 A 的方法体中确认存在对 B 的调用
3. 状态流转链（如 `INIT → CHARGED → PAID`）中的每个状态值，在枚举类中确认存在
4. 金额计算、核心业务规则描述，与代码实际实现一致

校验结果输出为清单：

```
| 链路描述 | 校验结果 | 失败原因 |
| --- | --- | --- |
| BillManager.createBill() → ChargeCalculator | ✅ 通过 | - |
| 状态 INIT → CHARGED | ❌ 失败 | 枚举 BillStatusEnum 中不存在 CHARGED，实际为 CHARGE_DONE |
```

**核心类完整性校验**：扫描阶段识别的该领域核心入口类，必须 100% 出现在核心文件表中。遗漏则补充。

**去重闭环**：若校验项 2（关键词重复）发现问题，**必须自动去重后重新校验**，而非仅输出建议。去重策略：保留索引中已有的关键词映射行，删除新增文件中产生的重复关键词，在输出中说明去重操作。

校验未全部通过时，修正后重新执行校验，直至全部通过或明确列出无法自动修复的问题交由用户处理。

**subagent 草稿的准确性验证（模式B）**：

若文档草稿由 subagent 生成，主流程必须对关键结论做抽样验证：
- 状态流转：至少抽样验证 1 条状态链的枚举值存在性
- 核心调用链：至少抽样验证 1 条链路的方法名和调用关系
- 高风险点：每个高风险点必须有具体行号引用，主流程抽检 1-2 个行号是否对应相关代码

---

#### 步骤7：更新协作约束文件引用路径

**首次构建时强制更新**：首次构建知识库时，**必须在协作约束文件追加"知识库入口"引用**（知识库目录路径 + 索引文件路径），确保后续任务能感知知识库存在。

**后续生成的更新条件**：
- 默认不更新协作约束文件，只更新知识库文件和索引
- 只有新增或调整长期稳定的协作规则、项目结构、模块职责、目录位置、外部依赖、业务流程引用规则时，才更新协作约束文件
- 若确需更新，必须先读取目标文件了解章节结构，再决定追加位置，不假设文件结构固定

---

#### 步骤8：更新知识库索引

在对应区块追加关键词映射行：
- 项目导航类 → 索引中项目导航类区块
- 工具/框架类 → 索引中工具/框架类区块
- 业务领域类 → 索引中业务领域类区块
- 数据模型类 → 索引中数据模型类区块

追加行时保持目标表的原有列名、列数和格式。

**关键词提取规则**：
- 项目导航类：概念名、模块名、子模块名
- 工具/框架类：框架名、工具名、配置键名、类名、接口名
- 业务领域类：类名、接口名、方法名（驼峰和下划线形式）、核心枚举名、业务概念中文名、数据表名
- 数据模型类：表名、DO 类名、核心字段名、枚举名

**关键词数量指引（解决量化标准缺失问题）**：

- 每个知识库文件的关键词建议 **10-25 个**
- 关键词应为"用户排查时会搜的词"（类名、表名、状态码、错误码、业务概念），**避免堆砌包名**
- 中英文都覆盖（中文业务概念 + 英文类名/方法名）

**关键词分隔符格式**：关键词之间用 ` / ` 分隔（前后含空格）；单个关键词内部不含空格；中文关键词与英文类名之间用 ` / ` 分隔。

示例：`发票开具 / InvoiceManager / createInvoice / 发票状态 / INVOICE_STATUS`

**去重闭环**：追加前先与索引已有行做精确匹配去重，确保同一关键词不会映射到多个知识库文件（除非确实是多文件交叉覆盖，此时在输出中说明）。

**批次进度更新**：每批生成完成后，同步更新索引顶部"待生成领域清单"的状态。

### 模式三：知识库体检（失效扫描）

当用户要求"体检知识库"/"扫描知识库失效引用"/"检查知识库是否过期"时执行。此模式解决知识库随代码演进而腐烂的问题。

#### 步骤1：读取知识库索引，收集所有引用

读取知识库索引，提取所有知识库文件路径。

#### 步骤2：扫描所有知识库文件，提取代码引用

对每个知识库文件，提取以下引用项：
- 文件路径（`com.example.xxx.YyyController` 形式或相对路径）
- 类名
- 方法名
- 枚举值 / 状态值
- 数据表名

#### 步骤3：与当前代码做 diff（优先使用工具）

**优先**使用 `validators/audit-knowledge.sh` 程序化扫描，将知识库中引用的类名、路径逐一在项目中 grep，输出未命中清单。

工具不可用时，AI 按以下规则手动扫描：
- 每个引用的类名 / 路径，在项目中搜索是否存在
- 每个引用的方法名，在对应类中搜索是否存在
- 每个引用的枚举值，在对应枚举类中搜索是否存在

#### 步骤4：输出失效报告

```

### 失效引用（代码中已不存在）
| 知识库文件 | 引用内容 | 类型 | 建议操作 |
| --- | --- | --- | --- |
| 账单全生命周期.md | BillManager.createBill() | 方法已删除 | 更新为 BillService.createBill() 或标注已废弃 |
| 账单全生命周期.md | ChargeBillDO | 类已移除 | 检查替代类并更新 |

| 知识库文件 | 引用内容 | 问题 | 建议操作 |
| --- | --- | --- | --- |

共 N 条引用校验通过。
```

列出失效报告后，询问用户："是否需要修复失效引用？"用户确认后，逐个修复并重新校验。

## 并行执行与 subagent 协作规范

中型及以上项目（> 200 个类或 > 3 个模块）的扫描和生成必须通过 subagent 并行执行，避免主流程上下文撑爆。本章节定义 subagent 协作的统一规范。

### 扫描阶段并行化（第一层 + 第二层）

**强制原则**：中型项目第一层/第二层扫描一律通过 subagent 执行，主流程只接收结构化摘要。禁止主流程直接全量读取类文件。

**模块划分**：按项目模块分配 subagent，每个 subagent 负责 1-2 个模块。

**并行规模**：建议 3-4 个 subagent 并行，不超过 5 个（避免资源竞争和结果合并复杂度）。

**subagent 输出统一格式**：

每个扫描 subagent 必须按以下结构输出，便于主流程合并：

```
## 模块扫描结果：{模块名}

### 模块概述
{模块职责简述}

### 领域归类清单
| 领域名称 | 涉及类 | 类职责简述 | 是否已被知识库覆盖 |
| --- | --- | --- | --- |
| {领域名} | {类名列表} | {简述} | 是/否 |

### 核心类清单
| 类名 | 类型 | 核心方法 | 业务含义 |
| --- | --- | --- | --- |
| {类名} | Controller/Service/Manager/DO/Enum | {方法签名} | {注释摘要} |
```

**subagent 结果校验与容错**：

并行 subagent 返回后，主流程必须校验：
1. 结果非空且包含上述结构化输出
2. 空结果或格式异常时，记录失败模块，在下一轮重试该模块，不阻塞其他模块
3. 重试仍失败，在输出中明确标注"模块 {X} 扫描失败，建议人工补充"
4. 主流程合并各 subagent 结果时，对跨模块的同名领域做去重或合并判断

### 生成阶段并行化（模式B 深度阅读 + 草稿生成）

当领域涉及类 > 10 时（步骤5 模式B），深度阅读和文档草稿生成委托 subagent 完成。

**subagent 任务定义**：
- 输入：领域名称 + 涉及类列表 + 对应模板
- 输出：符合模板结构的完整文档草稿

**准确性保障（解决最小读取与内容准确性的张力）**：

subagent 生成的草稿必须满足：
1. **代码依据标注**：文档中每个核心结论（状态流转、调用链、金额计算）必须标注代码依据（文件路径 + 行号区间）
2. **高风险点必须有行号**：每个高风险点必须包含具体位置（文件 + 行号），便于人工复核
3. **主流程抽样验证**：主流程对 subagent 草稿的关键结论做抽样验证（见步骤6"subagent 草稿的准确性验证"）

**subagent 草稿代码依据标注示例**：

```markdown
## 账单状态流转

- INIT → CHARGED（计费完成）→ SETTLED（出账完成）→ PAID（支付完成）
  依据：`BillStatusEnum.java:25-40`，`BillManager.transitionStatus():120-135`

## 高风险点

- [并发] 重复创建风险：syncBill 接口幂等校验缺失
  位置：`BillManager.java:85-92`
  影响场景：同一协议并发触发
  建议处理：增加基于协议号的分布式锁
```

## 知识库文件生成模板

四类知识库文件分别对应以下模板，生成时必须严格遵循对应模板的章节结构：

| 知识库类型 | 模板文件 | 当前充分适配的语言 |
| --- | --- | --- |
| 业务领域类 | [templates/业务领域类模板.md](templates/业务领域类模板.md) | Java / JVM 系 |
| 数据模型类 | [templates/数据模型类模板.md](templates/数据模型类模板.md) | Java / JVM 系 |
| 项目工具类 | [templates/项目工具类模板.md](templates/项目工具类模板.md) | Java / JVM 系 |
| 项目导航类 | [templates/项目导航类模板.md](templates/项目导航类模板.md) | 语言无关 |

> **多语言适配说明**：当前版本的业务领域类、数据模型类模板字段基于 Java 项目抽象。Python / Go / Node.js 等非 JVM 项目使用时，可将字段映射为对应概念（如 Python 的 View / Service / Model，Go 的 Handler / Service / Repository）。映射后的字段名写在原字段位置，保持表格结构不变。

> 模板内的占位符（`{xxx}`）需根据实际代码事实替换，禁止保留占位符原样输出。模板章节结构为最小集合，可在末尾追加领域专属章节，但不得删除既有章节。

## 项目协作约束文件更新规则

项目协作约束文件不是业务知识的默认承载位置，但**首次构建时必须追加知识库入口引用**。根据知识库文件类型，更新条件如下：

| 知识库类型 | 协作约束文件更新条件 |
| ------ | ------ |
| 首次构建（任意类型） | **强制更新**：追加知识库入口路径（目录 + 索引文件）到协作约束文件 |
| 项目导航类 | 项目结构与问题定位章节 |
| 工具/框架类 | 对应功能章节 |
| 业务领域类 | 默认不追加；仅当影响长期稳定协作规则时才补充 |
| 数据模型类 | 默认不追加 |

更新时必须先读取文件了解章节结构，不假设结构固定。历史文档只能作为辅助参考。

## 联动维护规则

### 维护规则内容

> 当任务涉及代码修改，且修改内容覆盖知识库已记录的模块、类、业务流程时，必须主动检查并同步更新对应知识库文档。

### 写入策略（草稿确认制，解决越权问题）

首次生成知识库后，需确保上述维护规则已存在于协作约束文件中。执行流程：

1. **检查**：协作约束文件中是否已包含上述维护规则
2. **已存在**：不做重复写入
3. **不存在**：**生成草稿内容**，输出建议落盘路径和草稿全文，**由用户确认后再写入**
   - 优先追加到既有 `.trae/rules/` 规则文件的合适章节末尾
   - 若无合适规则文件，建议新建 `.trae/rules/knowledge-maintenance.md`
   - 草稿必须遵循目标文件已有的格式风格
4. **写入确认**：用户明确同意后，才执行写入操作，并在输出中告知写入位置

**禁止**：未经用户确认直接写入或修改协作约束文件。

## 模糊输入提问规则

当用户输入无法精准识别时，**必须主动提问**，按以下维度：

1. **业务领域确认**：你指的是哪个业务领域？（列出可能的选项）
2. **覆盖深度确认**：你想要全链路知识还是特定子场景？
3. **关键词确认**：你提到的{关键词}在项目中有多个对应类，你指的是哪个？（列出具体类名和路径）

**绝对禁止**：自行猜测用户意图并直接生成。必须先确认再执行。

## 上下文保护规则

1. **分层渐进扫描**：第一层只扫描目录结构 → 第二层只读类注释和公开成员签名 → 第三层只在选定领域内做深度阅读
2. **分批生成**：每次最多生成 1-2 个知识库文件，3 个以上领域按优先级分批处理
3. **最小读取原则**：每个阶段只读取当前阶段所需的最少量文件
4. **禁止全量读取方法体**：识别未覆盖逻辑阶段只读类注释和签名
5. **批次量化**：第二层扫描每批 ≤ 20 个文件、每个文件 ≤ 80 行
6. **历史文档非权威**：历史需求/设计文档仅作辅助背景，知识库权威依据只能是当前代码
7. **中型项目强制 subagent**：> 200 个类或 > 3 个模块时，扫描和生成必须委托 subagent，主流程只接收结构化摘要
