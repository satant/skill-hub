# 知识库生成器（knowledge-generator）

本 skill 用于从项目代码中自动识别未被知识库覆盖的核心业务逻辑，生成知识库文件（含人类可读文档和 AI 可消费的机器可读摘要），同步更新知识库索引，并能反向扫描知识库中已失效的引用。

> **适用语言：Java / JVM 系（完整支持）、Vue / React 前端项目（v2.4.0 新增支持）**
>
> 通过语言配置文件（lang-profiles/）驱动脚本适配不同语言。Java 走原有完整流程，
> Vue/React 走适配后的扫描逻辑。Python/JS 签名提取三件套仍独立可用。

> **核心理念**：知识库的首要服务对象是 AI。目的是让 AI 在解决问题时有精准依据，减少每次扫描范围，避免遗漏细节。
>
> **v2.4.0 核心升级**：多语言支持（Java/Vue/React 通过语言配置文件驱动）。

## 触发条件

触发判定遵循**可操作规则**，不依赖主观判断词。

**生成触发（触发知识库生成/维护/体检流程）**：

1. 用户诉求中同时出现「动作词」+「沉淀意图词」：
   - 动作词：生成 / 补充 / 沉淀 / 写 / 维护 / 整理 / 梳理 / 扫描
   - 沉淀意图词：知识库 / 知识文档 / 后续查阅 / 长期留存
2. 用户明确要求"扫描项目生成知识库"/"自动生成知识库"/"扫描知识库缺口"
3. 用户明确给出想沉淀为知识库的逻辑方向，且语句中含"知识库""沉淀""留存"之一
4. 代码修改完成后，根据联动维护规则需主动更新知识库
5. 用户要求"体检知识库"/"扫描知识库失效引用"/"检查知识库是否过期"
6. 用户要求"继续生成知识库"/"继续扫描"，触发恢复模式

**查询消费触发（D3，不触发生成，触发"先查知识库"行为）**：

> 前提：项目已存在知识库（`$KB_ROOT/` 非空）。知识库不存在时此规则不生效。

7. 用户要求**理解/分析/设计**某个业务领域时：先查知识库，命中则基于知识库回答，未命中则正常分析
8. 用户要求**排查问题**时：先查知识库，命中则按知识库的排查建议和 grepHints 定位
9. 用户要求**修改代码**时：先查知识库，命中则按知识库的核心链路评估影响范围

查询消费触发不会启动生成流程，只是让 AI 先查再做事。

**不触发**：用户明确要求"设计文档""PRD""技术方案"而非"知识库"。

**模糊地带**：无法判定时，默认反问澄清。

## 通用异常处理

| 异常场景 | 处理策略 |
| --- | --- |
| 索引文件格式异常 | 终止当前步骤，输出异常片段 |
| 文件超大（> 5000 行） | 只读头部注释+方法签名 |
| 二进制/非 UTF-8 | 跳过 |
| 代码自相矛盾 | 以最后一次定义为准，标注冲突 |
| 工具不可用 | 降级为逐文件头部读取 |
| subagent 返回空/异常 | 记录失败模块，重试，不阻塞其他模块 |
| 校验脚本执行失败（权限/路径） | **不可降级**（v2.3.0）：修复脚本环境后重试，禁止跳过 |
| 校验脚本报错（校验不通过） | 强制修正后重新校验，禁止跳过 |

## 脚本执行门控机制（v2.3.0 新增，issue-006 核心修复）

> **核心原则**：配套脚本不是"可选增强"，而是"必须执行的检查门"。脚本未执行时，流程不可继续。

### 门控检查点

| 检查点 | 位置 | 检查条件 | 不满足时的动作 |
| --- | --- | --- | --- |
| **门控0：环境前置检查** | **步骤2 之前** | **python3 可用 + 脚本完整性 + 基础命令** | **自动安装 python3 后重试，不阻塞流程** |
| 门控1：A1/A2 脚本执行 | 步骤3 完成后 | A1（extract-business-terms.sh）和 A2（cluster-by-business-verb.sh）的输出文件存在 | 强制执行脚本后再进入步骤4 |
| 门控2：A6 大类枚举 | 步骤3 完成后 | extract-large-class-methods.sh 已执行，大类方法清单已产出 | 强制执行后再进入步骤4 |
| 门控3：A7 跨领域识别 | 步骤3 完成后 | cross-domain-scanner.sh 已执行，跨领域子场景登记表已产出 | 强制执行后再进入步骤4 |
| 门控4：B1 质量校验 | 步骤6 每个文件写入后 | validate-knowledge.sh 退出码为 0 | 强制修正后重新校验 |
| 门控5：B4 代码反向验证 | 步骤6 每个文件写入后 | cross-validate-with-code.sh 退出码为 0 | 强制修正后重新验证 |

### 门控0：环境前置检查与自愈（v2.3.0 新增）

> **设计原则**：门控机制防的是"偷懒跳过"，不是"环境缺失"。工具问题必须自愈，不阻塞流程。

**在所有其他门控之前**，必须先执行环境检查：

```bash
bash scripts/ensure-python3.sh
```

该脚本按顺序执行：
1. **python3 检测**：检查 `python3 --version` 是否可用
2. **自动安装**：不可用时按操作系统自动安装
   - macOS：通过 Homebrew 安装（无 Homebrew 时先安装 Homebrew）
   - Ubuntu/Debian：通过 apt-get 安装
   - CentOS/RHEL：通过 yum/dnf 安装
   - Alpine（容器）：通过 apk 安装
   - Windows（Git Bash/MSYS2/Cygwin）：通过 winget/choco/scoop 安装
3. **基础命令校验**：grep / awk / sed / find / wc 是否可用
4. **脚本完整性校验**：所有门控脚本和校验器文件存在且可读

**自愈优先，阻塞兜底**：先尝试自动修复环境，只在自动安装失败时才提示用户手动安装。

### 脚本执行规则

1. **不可跳过**：门控检查点标注的脚本必须执行，AI 不得以"已通过阅读代码获得足够信息"为由跳过
2. **不可降级**：脚本执行失败时，修复脚本环境后重试，禁止降级为 AI 手动校验
3. **执行证据**：脚本输出文件必须保存到 `$KB_ROOT/.cache/` 目录，作为执行证据
4. **RunCommand 调用**：AI 必须通过 RunCommand 工具执行脚本，不可用 Grep/Read 替代
5. **环境自愈**（v2.3.0）：门控0 在所有其他门控之前执行，python3 缺失时自动安装

### 脚本输出消费方式

```bash
# 门控0：环境前置检查（最先执行）
bash scripts/ensure-python3.sh

# v2.4.0 项目语言检测（门控0 之后、A1 之前执行）
# 自动检测项目语言，获取对应的语言配置文件路径
LANG_PROFILE=$(bash lang-profiles/detect-language.sh <项目根目录>)
# 后续脚本通过 LANG_PROFILE 环境变量自动适配语言

# A1 业务术语提取（自动适配语言）
LANG_PROFILE="$LANG_PROFILE" bash scripts/extract-business-terms.sh <源码目录> $KB_ROOT/.cache/business-terms.txt

# A2 业务方法三维聚类（自动适配语言）
LANG_PROFILE="$LANG_PROFILE" bash scripts/cluster-by-business-verb.sh <源码目录> $KB_ROOT/.cache/verb-clusters.txt

# A6 大文件方法枚举（阈值按语言自动调整：Java 500/15，前端 300/10）
LANG_PROFILE="$LANG_PROFILE" bash scripts/extract-large-class-methods.sh <源码目录> $KB_ROOT/.cache/large-class-methods.txt

# A7 跨领域子场景识别（自动适配语言）
LANG_PROFILE="$LANG_PROFILE" bash scripts/cross-domain-scanner.sh <源码目录> $KB_ROOT/.cache/cross-domain-scenarios.txt

# B1 质量校验（每个文件写入后执行，自动适配语言）
LANG_PROFILE="$LANG_PROFILE" bash validators/validate-knowledge.sh <知识库文件.md> <项目根目录> [索引文件.md] --json

# B4 代码反向验证（每个 .summary.json 写入后执行，自动适配语言）
LANG_PROFILE="$LANG_PROFILE" bash validators/cross-validate-with-code.sh <summary.json 文件> <项目源码目录> --json
```

## 工作流程

### 模式一：自动扫描

#### 步骤1：协作约束文件与知识库入口前置检查

**1.1 检测协作约束文件**

优先级：`.trae/rules/` > `AGENTS.md` > `CLAUDE.md` > `.cursorrules` > `README.md`

**降级（两阶段填充机制 v2.4.0）**：无协作约束文件时，在项目根目录创建 `AGENTS.md`（参照 [templates/AGENTS.md](templates/AGENTS.md)），告知用户后继续。创建时分两阶段填充：

**阶段1：创建时快速填充**（步骤1.1 执行时）

基于项目根目录的快速扫描（≤ 30秒），填充以下内容：
1. **项目概述**：从 README.md / package.json / pom.xml 提取项目名称和描述
2. **技术栈**：从构建配置文件提取（Java: pom.xml/build.gradle，前端: package.json dependencies）
3. **项目语言**：执行 `detect-language.sh` 获取语言类型
4. **项目结构**：扫描一级和二级目录，识别模块划分 + 目录职责说明 + 分层依赖方向
5. **编码规范**：根据项目语言保留对应段落（Java 后端编码规范 / 前端编码规范），删除另一组
6. **工作流**：根据项目语言保留对应段落（Java 后端工作流 / 前端工作流），删除另一组
7. **知识库路径**：填入 `$KB_ROOT` 值
8. **构建/启动命令**：从构建配置提取（如 `mvn` / `npm run build`）

> **前后端裁剪规则**：阶段1 根据 `$LANG_PROFILE` 的 `LANG_NAME` 裁剪模板。Java 项目删除「前端编码规范」和「前端工作流」段落，前端项目删除「Java 后端编码规范」和「Java 后端工作流」段落。

**阶段2：扫描后回填**（步骤4 用户确认领域后，步骤5 生成知识库文件时）

基于步骤3 的深度扫描结果，回填以下内容：
1. **核心业务领域速览**：从扫描到的领域清单回填（领域名/入口/知识库文档路径）
2. **核心文件速查**：从步骤3 识别的核心入口类/文件回填
3. **问题定位**：从步骤3 识别的常见问题排查路径回填
4. **Grep 快速定位**：补充按业务场景的快速搜索命令

> **关键原则**：阶段1 创建的是"可用骨架"，阶段2 回填的是"实质内容"。不允许阶段1 就跳过通用规则，也不允许阶段2 才补通用规则。两阶段完成后 AGENTS.md 应能独立指导 AI 协作。

**1.2 检查上下文注入**

- Trae 项目：检查 `.trae/rules/` 是否存在
- 非 Trae：告知用户需在 IDE 中配置 `AGENTS.md` 为始终包含

**1.3 扫描协作约束文件，提取知识库位置**

- 扫描协作约束文件（AGENTS.md / .trae/rules/ 等），查找其中记录的知识库入口路径
- 如果约束文件中有知识库路径（如 `知识库入口：docs/wiki/知识库索引.md`），提取该路径作为 `$KB_ROOT` 的首选值
- 如果约束文件中无知识库路径，继续执行步骤1.4 的自动探测
- 历史文档仅作辅助背景，**不能作为权威依据**

**1.4 知识库根目录探测（v2.3.0 新增）**

按以下优先级确定知识库根目录 `$KB_ROOT`：

| 优先级 | 探测方式 | 示例 |
| --- | --- | --- |
| 1 | 用户显式指定 | 用户说"知识库在 docs/wiki/" |
| 2 | 协作约束文件中的知识库入口路径 | AGENTS.md 中记录的路径 |
| 3 | 自动扫描常见位置（含索引文件的目录） | `docs/knowledge/`、`docs/wiki/`、`knowledge-base/`、`docs/architecture/` |
| 4 | 以上均未找到 | 默认使用 `docs/knowledge/`，告知用户 |

> **关键原则**：`$KB_ROOT` 一旦确定，后续所有步骤的路径都基于此变量拼接。不硬编码 `docs/knowledge/`。
> 探测到已有知识库时，优先使用已有路径，不在新位置创建重复知识库。

---

#### 步骤2：扫描现有知识库覆盖范围

**首次构建分支**：`$KB_ROOT/` 目录或索引不存在时，一次性创建目录 + 六分区表头（项目导航 / 项目工具 / 业务领域 / 数据模型 / 外部依赖 / 架构专题），并在协作约束文件中强制追加知识库入口引用。

**增量更新分支（v2.3.0 新增）**：`$KB_ROOT/` 目录已存在且索引非空时，执行增量评估：

1. **读取 progress.json**：检查 `$KB_ROOT/.cache/progress.json`，获取已完成的领域清单和已扫描类清单
2. **读取现有索引**：解析索引中所有关键词和文档路径，构建"已覆盖集合"
3. **增量变更检测**：对比 progress.json 中的 scannedClasses 与当前代码库的差异（新增类/删除类/修改类）
4. **输出增量评估报告**：
   - 已覆盖且无变更的领域 → 标记为"跳过"，不进入步骤3 扫描
   - 已覆盖但有新增类的领域 → 标记为"待补充"，只扫描新增类
   - 完全未覆盖的领域 → 标记为"新建"，进入步骤3 正常流程
5. **跳过优化**：如果 80% 以上领域已覆盖且无变更，告知用户"知识库覆盖率已较高，建议使用模式二定向补充"

> **关键原则**：增量更新时，已有文档的领域不做全量重扫，只针对变更部分增量补充。这是避免"每次都从头扫描"的核心优化。

**覆盖判定规则**：完全匹配才算覆盖（类名/概念/表名/方法名/模块名）。模糊语义相似不算。宁可多报候选。

---

#### 步骤3：分层渐进扫描识别未覆盖逻辑（A1-A8 增强识别）

**中型项目（> 200 个类或 > 3 个模块）必须委托 subagent 并行扫描。**

**增量更新场景优化（v2.3.0 新增）**：
- 如果步骤2 判定为增量更新（非首次构建），门控脚本的执行范围按以下规则缩减：
  - **"跳过"领域**：不执行任何脚本，不进入步骤3
  - **"待补充"领域**：只对新增/变更的类执行 A6 大类枚举和 A2 聚类，不重扫全项目
  - **"新建"领域**：正常执行所有门控脚本（A1/A2/A6/A7）
- 门控检查点仍然生效，但检查对象从"全项目"变为"本次扫描范围"

**第一层：目录结构扫描**

只扫描包名和类名。输出各模块类名清单。

**第二层：摘要式扫描**

只读类注释和公开成员签名。优先用脚本提取，降级按每批 ≤ 20 文件/≤ 80 行读取。

**增强识别流程（A1-A8，在第二层扫描后、第三层前执行）**：

> 以下 8 个增强项在第二层摘要式扫描完成后、用户选定领域前执行，构成完整的"增强领域识别"流程。
> 脚本参数：`<源码目录>` 为项目根目录下的 Java 源码目录（如 `src/main/java`），`[输出文件]` 为可选的输出路径。
> **门控约束**：A1/A2/A6/A7 的脚本必须执行（参见"脚本执行门控机制"），否则步骤3 不可完成。

**A1 业务术语候选提取**：

- 执行 `bash scripts/extract-business-terms.sh <源码目录> [输出文件]` 提取含中文字符串的候选清单
- 脚本只负责提取"候选字符串"，AI 对候选做分词和归类，形成业务术语字典
- 字典格式：`{中文术语, 英文对应, 典型出现位置}`

**A2 业务方法三维聚类（v2.3.0 升级，issue-005）**：

- 执行 `bash scripts/cluster-by-business-verb.sh <源码目录> [输出文件]` 按三维聚类方法名
- **维度1：动词聚类**（原有）- open/apply/check/build/query 等
- **维度2：业务名词聚类**（新增）- Red/RedRush/Invalid/Invoice 等，识别同一业务族
- **维度3：修饰词变体标注**（新增）- Part/Fast/Direct/ISV/Pre 等，区分业务变体
- 聚类结果用于识别同一业务领域的跨模块方法集合和业务变体
- 高频出现的"动词+名词"组合是领域边界的强信号

**A3 方法级重叠率判定**：

- 对候选领域计算方法级重叠率：共享方法名占两个领域总方法数的比例
- 重叠率 ≥ 50%：应合并为一个领域
- 重叠率 30%-50%：考虑合并或建立主从关系
- 重叠率 < 30%：独立领域

**A4 覆盖深度校验（v2.3.0 增强，issue-004）**：

- 对每个已有知识库文档检查**覆盖密度 + 覆盖率**双维度：

| 维度 | 校验项 | 阈值 | 不通过处理 |
| --- | --- | --- | --- |
| 密度（原有） | 链路描述行数/条数 | ≥ 10行/条 | 要求补充描述 |
| 方法覆盖率（新增） | 核心入口类方法覆盖率 | ≥ 70% | 要求补充阅读 |
| 变体覆盖（新增） | 业务变体链路覆盖率 | 100% | 要求补充变体链路 |
| 状态覆盖（新增） | 状态枚举在文档中出现率 | ≥ 90% | 要求补充状态说明 |

**A5 跨模块领域重组**：

- 当 A1-A4 识别出的领域跨越多个模块时，按以下聚合规则重组：
  - 聚合依据1：业务术语字典命中同一概念的方法集合
  - 聚合依据2：跨模块调用链所经类（Controller→Service→DAO 完整链路）
  - 聚合依据3：共享枚举状态的类集合（如都操作同一状态枚举的类）

**A6 大类强制方法枚举（v2.3.0 新增，issue-001）**：

> **解决问题**：大类（God Class）中 40% 的子场景被系统性遗漏，如 2000+ 行的 InvoiceManager 中部分冲红方法（第888-1100行）从未被读取。

- 执行 `bash scripts/extract-large-class-methods.sh <源码目录> [行数阈值] [方法数阈值] [输出文件]`
- 默认阈值：行数 > 500 或 public 方法数 > 15
- 脚本产出：每个大类的完整方法清单 + 按动词聚类 + 业务变体识别提示
- **AI 必须消费**：对每个大类，为所有 public 方法标注业务子场景（而非只读前 N 个方法）
- 类尾部方法（行号 > 500）是遗漏高发区，必须检查是否已被领域文档覆盖
- 方法覆盖率 < 80% 的大类，不允许标记领域为「已完成」

**A7 跨领域子场景识别（v2.3.0 新增，issue-002）**：

> **解决问题**：跨领域子场景被"双重遗漏"——两个领域都认为"不属于我"而跳过。

- 执行 `bash scripts/cross-domain-scanner.sh <源码目录> [输出文件]`
- 脚本产出：跨模块入口方法清单 + 跨模块调用链 + 高频状态枚举引用
- **AI 必须输出跨领域子场景登记表**：

```markdown
| 子场景 | 入口领域 | 完整链路领域 | 入口方法 | 关联状态枚举 |
| --- | --- | --- | --- | --- |
| 部分冲红 | 发票冲红作废 / 发票开具主链路 | 发票冲红作废 | applyPartRedRush | PART_RED_RUSHED |
```

- 跨领域子场景的归属规则：
  1. 选择一个"主领域"记录完整链路
  2. 在关联领域记录入口引用和"参见"指针
  3. 在知识库索引中增加"子场景路由表"
  4. 未被任何领域认领的子场景必须补充或独立成新领域

**A8 业务变体识别（v2.3.0 新增，issue-005）**：

> **解决问题**：同一业务的不同变体（完整冲红/部分冲红/快捷冲红）被当作独立方法，无法关联。

- 基于 A2 的维度2（业务名词族）+ 维度3（修饰词标注），识别业务变体
- 对每个业务族，按修饰词分组识别变体：
  - 无修饰词 → 默认/完整变体
  - Part/Partial → 部分变体
  - Fast → 快捷变体
  - Direct → 自营变体
  - ISV → ISV变体
  - Pre → 预操作变体
- 对每个变体，检查是否有独立的：Controller 入口、调用链、状态流转
- 输出"业务变体清单"，作为 subagent 深度阅读的必读项
- subagent 必须为每个变体追踪独立调用链，在文档中写独立的"核心链路"

**业务领域划分原则**：
- 至少 3 个强相关核心类
- 对应可独立排查的业务场景
- 领域间核心类重叠率 < 30%
- 单文档 ≤ 300 行
- **跨领域子场景必须在两个领域都登记**（v2.3.0 新增）

**第三层：深度代码阅读（用户选定领域内）**

用户确认后，只对选定领域做深度阅读。涉及类 > 10 时委托 subagent（模式B）。

**大类深度阅读特殊规则（v2.3.0 新增）**：
- 对 A6 识别出的大类，subagent 必须读取所有 public 方法（而非只读前 N 行）
- 大类阅读不受"每文件 ≤ 80 行"的批次量化限制，必须分批完整阅读

**领域标记"已完成"前置条件（v2.3.0 新增，issue-004）**：

标记领域为"已完成"前，必须检查：
1. 核心子场景待补充条目 ≤ 2 条
2. 核心入口类的 public 方法覆盖率 ≥ 70%
3. 已识别的业务变体全部有对应链路描述
4. 状态枚举值在状态流转章节中全部出现
5. .summary.json 的 coverageGaps 中无 gapType=core 的缺口

不满足条件 1-3 时，强制补充后再标记完成。

**"待补充"章节与 coverageGaps 同步规则（v2.3.0）**：

"待补充"是可选章节，不作为模板的永久结构。规则如下：
1. **单一数据源**：coverageGaps（.summary.json）是未覆盖场景的唯一数据源
2. **同步生成**：.md 的"待补充"章节内容从 coverageGaps 同步生成，不独立维护
3. **可选保留**：coverageGaps 为空时（领域已充分覆盖），删除 .md 中的"待补充"章节，不保留空壳
4. **分类对应**：coverageGaps 中 gapType=core 对应"核心子场景"子节，gapType=detail 对应"实现细节"子节

**深度阅读分流**：涉及类 ≤ 10 主流程直读（模式A）；> 10 委托 subagent（模式B）。

**双产物机制（D2 核心落地）**：

每个知识库文档生成时，**同时产出两个文件**：

```
$KB_ROOT/业务领域/发票冲红/
├── 发票冲红主链路.md           # 人类可读视图
└── 发票冲红主链路.summary.json # AI 可消费的首要产物
```

**生成顺序**：先完成 .md 文档（深度阅读→组织内容→写入），然后基于 .md 内容提取结构化字段生成 .summary.json。两者在同一步骤中完成，不跨批次。

**.summary.json 是 AI 查询的首选入口**。生成规则：
- 严格遵循 [templates/机器可读摘要模板.json](templates/机器可读摘要模板.json) 结构
- `entryPoints`：入口类和方法列表（AI 定位代码的起点）
- `coreChain`：核心调用链（AI 理解业务流程的骨架）
- `stateMachine`：状态枚举和流转规则（AI 排查状态问题的依据）
- `grepHints`：可直接执行的搜索命令（AI 快速定位代码的工具）
- `confidence`：每条信息的置信度标注（high/low，low 项需说明原因）
- `coverageGaps`：已知未覆盖的子场景（AI 知道哪些地方需要额外扫描）

**文件放置规则**：
- 项目导航类 → `$KB_ROOT/` 根目录
- 项目工具类 → `$KB_ROOT/项目工具/`
- 业务领域类 → `$KB_ROOT/业务领域/{领域名称}/`
- 数据模型类 → `$KB_ROOT/数据模型/`
- 架构专题类 → `$KB_ROOT/架构专题/`（v2.3.0 新增）
- 外部依赖 → 索引文件底部"外部依赖清单"表格维护，无独立目录

#### 步骤6：质量校验（B1 不可降级 + B4 代码反向验证）

**B1 强制执行点（v2.3.0 不可降级）**：

```bash

```

- 脚本**执行失败**（权限/路径问题）：**不可降级**（v2.3.0），修复脚本环境后重试
- 脚本**校验不通过**（exit code 1）：**强制修正后重新校验，禁止跳过**
- 支持 `--json` 参数输出结构化报告（便于 AI 消费）
- **禁止用 AI 手动 Read 检查替代脚本执行**（issue-009 确认偏差修复）

**B4 代码反向验证（v2.3.0 新增，issue-009）**：

对每个 .summary.json，**必须执行** `validators/cross-validate-with-code.sh`：

```bash
bash validators/cross-validate-with-code.sh <summary.json 文件> <项目源码目录> --json
```

反向验证项：
1. **枚举值验证**：提取 stateMachine.states 的所有 value，去代码中验证是否存在
2. **方法名验证**：提取 coreChain 中提到的所有方法名，去代码中验证是否存在
3. **类名验证**：提取 coreClasses 中的所有 className，去代码中验证路径是否正确
4. **路径验证**：提取所有 filePath，验证文件是否存在，检查是否包含 `...` 缩写
5. 退出码非 0 时，强制修正后重新验证

**程序化校验项**：

| 序号 | 校验项 | 说明 |
| --- | --- | --- |
| 1 | 路径存在性（B2增强） | 支持多种路径格式：Java 类名、反引号路径、表格裸路径、含 src/main 的片段 |
| 2 | 关键词重复 | 新增关键词不与索引重复 |
| 3 | 索引格式 | 保持原有列名、列数 |
| 4 | 交叉引用 | .md 引用路径正确 |
| 5 | 模板字段完整 | 覆盖必填章节 |
| 6 | **AI 可消费性（B3）** | .summary.json 存在且关键字段（domain/entryPoints/coreClasses/coreChain/stateMachine/grepHints/coverageGaps）非空 |
| 7 | **覆盖深度双维度（v2.3.0）** | 密度（≥10行/条）+ 方法覆盖率（≥70%）+ 待补充分类检查 |
| 8 | **代码反向验证（B4）** | 枚举值/方法名/类名/路径在代码中存在 |

**核心链路回溯校验（AI 执行）**：

对每条"入口→关键逻辑"调用链，反向在代码中验证：
1. 方法名存在
2. 调用关系成立
3. 状态值在枚举中存在
4. 核心业务规则与代码一致

**subagent 草稿验证（模式B，v2.3.0 增强）**：

草稿由 subagent 生成时，主流程执行**双重验证**：
1. **准确性验证**（原有）：抽样验证状态链枚举值、调用链方法名、高风险点行号
2. **完整性验证**（新增）：检查 subagent 的覆盖率自检报告，方法覆盖率 < 70% 时要求补充阅读

**核心类完整性**：扫描阶段识别的核心入口类必须 100% 出现在核心文件表。

**去重闭环**：关键词重复必须自动去重后重新校验。

### 模式四：外部依赖补充（C1）

当知识库文档中存在"待二方库确认"的标注时，可触发此模式补充外部依赖的结构信息。

**合规优先原则**：

1. **LICENSE 检查**：执行 `scripts/decompile-external-jar.sh`，脚本会先检查 jar 的 LICENSE/NOTICE 文件
   - 标注"禁止反编译"→ 跳过，在文档中标注"LICENSE 限制，需人工确认"
   - 标注"允许"→ 使用 javap 提取结构信息

4. **坐标定位**：从 pom.xml 解析依赖坐标定位 jar，不从类名反推

**输出**：
- 结构信息文件（`struct-extract.json`）
- 合规报告（`COMPLIANCE.md`）
- 在知识库文档和 .summary.json 的 `externalDependencies` 字段中更新

## AI 可消费性设计（核心章节）

> 本章节定义知识库作为"AI 解决问题的精准依据"的设计原则，是 v2.2.0 的核心理念落地。

### 查询优先原则（D1 + D3 落地）

**AI 执行任何任务前，应先查询知识库**（软约束，依赖 AGENTS.md 被注入上下文）。通过以下机制实现：

**1. 查询工具**：

```bash
bash scripts/query-knowledge.sh <知识库目录> <关键词1> [关键词2] ...
```

输出 JSON，包含 `matched`、`results`（含 summaryFile 路径）、`hint`（消费建议）。

**2. 三阶段消费+回退流程（v2.3.0 增强，issue-010）**：

```
用户任务
  │
  ▼
阶段1：索引查询（≤ 5秒）
  查询知识库索引，匹配关键词。
  │
  ▼
阶段2：命中质量评估（≤ 3秒）
  对索引命中的结果，评估 .summary.json 的信息质量：
  ┌──────────────┬──────────────────────────────────────┬──────────────────────────────┐
  │ 质量等级     │ 判定条件                             │ 处理方式                     │
  ├──────────────┼──────────────────────────────────────┼──────────────────────────────┤
  │ 充分         │ 有≥1条coreChain + ≥1个入口方法       │ 消费知识库，按grepHints定位  │
  │ 部分         │ 有coreClasses但无coreChain           │ 消费方向信息，补充Grep代码   │
  │ 不足         │ 只有businessTerms或状态枚举碎片      │ 立即回退到全量Grep代码       │
  │ 未命中       │ 索引无匹配                           │ 直接全量Grep代码             │
  └──────────────┴──────────────────────────────────────┴──────────────────────────────┘
  │
  ▼
阶段3：回退执行（质量为"不足"或"未命中"时）
  - 声明"知识库覆盖不足，回退到代码搜索"
  - 记录未覆盖场景到 $KB_ROOT/.cache/missed-queries.log
  - 直接执行 Grep/SearchCodebase 定位代码
  - 不在知识库中反复搜索浪费时间
```

**3. 知识库覆盖反馈闭环（v2.3.0 新增，issue-010）**：

当 AI 发现知识库覆盖不足时，必须记录反馈：

```bash
# 在 $KB_ROOT/.cache/missed-queries.log 中追加记录
echo "$(date +%Y-%m-%d) | {查询场景} | {命中领域}(不足/未命中) | 回退到Grep | {Grep命中文件数}个文件" \
  >> $KB_ROOT/.cache/missed-queries.log
```

- 累计 3 次以上同类未命中时，主动提示用户："检测到'{场景}'已被多次查询但知识库覆盖不足，建议补充该子场景的知识库文档"
- 用户可选择立即补充（触发模式二）或在下次知识库更新时统一补充
- 此机制使知识库的覆盖率随使用而自愈提升

**4. AGENTS.md 强约定**：

在协作约束文件中写入知识库消费规则（含三阶段回退），确保每次 AI 会话都知道要查询知识库及未命中时的处理方式：

```markdown
## 知识库消费规则

> 执行任何任务前，必须先查询知识库索引（`$KB_ROOT/知识库索引.md`）。
> 命中后优先消费 .summary.json，按 grepHints 定位代码，减少扫描范围。
> 未命中或命中质量不足时，立即回退到 Grep/SearchCodebase，不在知识库中反复搜索。
> 未命中场景必须记录到 `.cache/missed-queries.log`，供后续知识库增量补充。
```

**5. 场景触发（D3）**：

- 用户要求"理解 XX 业务"/"分析 XX 链路"/"设计 XX 方案"时：先查知识库，命中则基于知识库内容回答
- 用户要求"排查 XX 问题"时：先查知识库，命中则按知识库的排查建议和 grepHints 定位
- 用户要求"修改 XX 代码"时：先查知识库，命中则按知识库的核心链路评估影响范围
- 以上场景知识库未命中或命中质量不足时：**立即回退到 Grep/SearchCodebase**，不浪费时间

### 双产物机制（D2 详细规范）

**生成规则**：
- .summary.json 与 .md **同级生成**，不是可选附加
- .summary.json 的 `docFile` 字段指向对应的 .md 文件
- .md 文件不再需要内嵌结构化数据（JSON 承担了这个角色）

**维护规则**：
- .md 文件修改后，必须同步更新 .summary.json
- 体检模式同时检查两者的一致性

**消费优先级**：
1. AI 先读 .summary.json（结构化、解析快、信息密度高）
2. 仅在需要深入细节（如完整的方法体分析）时读 .md
3. grepHints 中的命令可直接执行定位代码

### 评分卡的 AI 维度（B3）

文档质量评分卡新增"AI 可消费性"维度（15分）：

| 维度 | 分值 | 评分标准 |
| --- | --- | --- |
| 机器可读摘要存在且完整 | 5 | .summary.json 存在，entryPoints/coreChain 非空 |
| 结构化字段可机器解析 | 5 | grepHints 是可执行命令数组 |
| 置信度标注清晰 | 5 | confidence 区分 high/low，low 项有说明 |

**领域复杂度弹性**：涉及类 < 5 的简单领域，评分门槛从 80 降至 65。

## 并行执行与 subagent 协作规范

**扫描阶段并行化**：中型项目第一层/第二层扫描通过 subagent 执行，主流程只接收结构化摘要。

**subagent 输出统一格式**：模块扫描结果 + 领域归类清单 + 核心类清单 + **覆盖率自检报告**（v2.3.0）。

**subagent 结果校验与容错**：空结果/异常重试，不阻塞其他模块。

**生成阶段并行化（模式B）**：领域涉及类 > 10 时，委托 subagent 完成深度阅读+草稿生成。草稿必须标注代码依据（文件路径+行号区间）。

### subagent 深度阅读量化指令模板（v2.3.0 新增，issue-003）

> **解决问题**：subagent 阅读深度不可控，不同领域覆盖深度 55%-85% 不等。

每次委托 subagent 时，必须包含以下标准化指令：

```markdown
你是知识库生成助手，请对「{领域名称}」领域进行深度阅读。

## 必须完成的操作

1. **全方法枚举**：对每个核心类，列出所有 public 方法（不只是前 N 个）
   - 参照 A6 大类方法枚举报告，确保大类（>500行/>15方法）的每个方法都被覆盖
2. **业务变体识别**：识别同一业务的所有方法变体
   - 参照 A2 三维聚类的维度2+维度3，识别业务名词族和修饰词变体
   - 每个变体必须追踪其独立调用链
3. **调用链追踪**：对每个入口方法，追踪调用链至少 3 层
   - Controller → Service → Manager → DAO
   - 或 Facade → Processor → Manager → DAO
4. **方法重载识别**：识别同名方法的所有重载，标注参数差异
5. **状态枚举引用扫描**：识别该领域涉及的所有状态枚举值，列出引用这些值的类清单
6. **大类尾部方法检查**：对行号 > 500 的方法，必须检查是否属于核心子场景

## 输出格式要求

除标准输出外，必须包含覆盖率自检报告（写入 .summary.json 的 coverageReport 字段）：
- totalPublicMethods：核心入口类的 public 方法总数
- readMethods：已读取/覆盖的方法数
- methodCoverage：方法覆盖率百分比
- unreadMethods：未读方法列表及原因
- identifiedVariants：已识别的业务变体清单
- tracedChains：已追踪的调用链清单（含层数）

## 主流程校验

- 方法覆盖率 < 70%：要求 subagent 补充阅读
- 存在未识别变体：要求 subagent 重新扫描
- 调用链深度 < 2：要求 subagent 加深追踪
```
