## 编码规范

### 通用规范

- **命名规范**：
  - 类/组件命名：{Java: 大驼峰 / 前端: 大驼峰组件}
  - 方法/函数命名：小驼峰，动词开头
  - 常量命名：全大写下划线，如 `MAX_RETRY_COUNT`
- **注释规范**：
  - 公共方法必须有注释，说明「为什么」而非「做什么」
  - 复杂业务逻辑必须有行内注释说明决策原因
  - TODO 必须带责任人标记：`// TODO(姓名) 描述`

### Java 后端编码规范

{阶段1：扫描项目现有代码风格后回填，以下为通用基线}

- **分层规范**：
  - Controller 只做参数校验和转发，不含业务逻辑
  - Service 处理业务编排，Manager 处理通用逻辑
  - DAO 只做数据操作，不包含业务判断
- **类后缀约束**：{根据扫描结果回填，如 Controller/Service/Manager/Repository/DO/DTO}
- **异常处理**：业务异常继承统一基类（如 `BizException`），禁止捕获后静默吞掉
- **日志规范**：
  - 入口层（Controller）记录请求摘要（INFO）
  - 业务层（Service/Manager）记录关键决策点（INFO）和异常（ERROR）
  - 数据层（DAO）不记录日志（由框架统一处理）
  - 禁止用 `e.printStackTrace()`，必须用 `log.error("描述", e)`
- **事务规范**：`@Transactional` 只加在 Service 层，不加在 Controller/DAO
- **依赖注入**：{扫描后确认是构造器注入还是字段注入，回填项目约定}

### 前端编码规范

{阶段1：扫描项目现有代码风格后回填，以下为通用基线}

- **组件规范**：
  - 单文件组件（SFC）结构顺序：`<template>` → `<script>` → `<style>`
  - 组件名必须多单词（避免与 HTML 元素冲突），如 `OrderList` 而非 `Order`
  - props 必须定义类型和默认值
- **状态管理规范**：
  - 全局状态用 Store（Pinia/Vuex/Redux/Zustand），局部状态用 ref/reactive/useState
  - Store 按业务域拆分，不建单一巨型 Store
  - 异步操作（API 调用）放在 Store action 或 Hook 中，不放在组件内
- **API 调用规范**：
  - 所有 API 请求统一封装在 `api/` 或 `services/` 目录
  - 请求函数必须定义请求参数和响应的 TypeScript 类型
  - 禁止在组件中直接用 `fetch`/`axios`，必须通过封装的 API 模块
- **样式规范**：{扫描后确认是 CSS Modules / Scoped CSS / Tailwind / Styled Components，回填项目约定}
- **错误处理**：
  - API 错误统一在请求拦截器中处理（全局 toast/通知）
  - 组件级错误用 ErrorBoundary（React）或 errorCaptured（Vue）捕获
  - 禁止 `.catch(() => {})` 静默吞掉错误

## 工作流

> 以下工作流按项目语言区分。Java 后端和前端项目各有独立的工作流规范。
> 根据项目语言保留对应段落，删除另一组。

### Java 后端工作流

#### 分析阶段

1. **先查知识库**：按"知识库消费规则"查询，了解现有业务逻辑和影响范围
2. **定位代码**：从 Controller → Service → Manager → DAO 完整追踪调用链
3. **识别影响范围**：
   - 检查涉及的数据库表和 DO/DTO
   - 检查涉及的枚举状态流转
   - 检查涉及的消息队列/定时任务/事件通知
4. **确认约束**：
   - 是否有事务边界要求
   - 是否有并发控制要求（分布式锁/乐观锁）
   - 是否有幂等性要求

#### 设计阶段

1. **接口设计**：
   - 先设计请求/响应 DTO，定义字段和校验注解
   - Controller 方法签名遵循 RESTful 规范
   - 复杂查询参数用 Query 对象封装，不用过多 `@RequestParam`
2. **分层设计**：
   - 明确每一层的职责：Controller（校验转发）→ Service（业务编排）→ Manager（通用逻辑）→ DAO（数据操作）
   - 跨模块调用通过 Service 接口，不直接调用其他模块的 DAO
3. **数据模型设计**：
   - 新增字段必须同步更新 DO 和 DTO
   - 枚举值变更必须评估所有引用点
   - 数据库变更必须有 DDL 脚本
4. **异常设计**：
   - 定义业务异常类，明确错误码和错误消息
   - 不在 DAO 层抛业务异常

#### 实现阶段

1. **编码顺序**：DTO/DO → DAO → Service → Controller → 单元测试
2. **分层编码约束**：
   - Controller：只做参数校验（`@Valid`）、调用 Service、组装响应
   - Service：业务逻辑编排，事务边界控制，异常转换
   - Manager：通用业务逻辑（可被多个 Service 复用）
   - DAO：数据库操作，不含业务判断
3. **测试规范**：
   - Service 层必须有单元测试，覆盖核心业务分支
   - 测试类命名：`{ClassName}Test`
   - 测试方法命名：`{method}_{scenario}_{expectedResult}`，如 `createOrder_success_returnsOrderId`
4. **提交规范**：
   - 单次提交只包含一个功能点或一个 bugfix
   - 提交信息格式：`{type}: {description}`，type 为 feat/fix/refactor/test/docs

### 前端工作流

#### 分析阶段

1. **先查知识库**：按"知识库消费规则"查询，了解现有页面和组件结构
2. **定位代码**：从路由 → 页面 → 组件 → Store/Hook → API 完整追踪数据流
3. **识别影响范围**：
   - 检查涉及的页面路由和导航
   - 检查涉及的组件层级和 props 传递
   - 检查涉及的 Store 状态和 API 调用
   - 检查涉及的 TypeScript 类型定义
4. **确认约束**：
   - 是否涉及权限控制（路由守卫/按钮权限）
   - 是否涉及 SEO 要求（影响 SSR/CSR 选择）
   - 是否涉及国际化（i18n）

#### 设计阶段

1. **组件设计**：
   - 先画组件树，明确 Container（有逻辑）和 Presentational（纯展示）的边界
   - 复用组件提取到 `components/`，页面专属组件留在 `pages/xxx/components/`
   - props 设计：数据向下，事件向上（单向数据流）
2. **状态设计**：
   - 全局状态（用户信息/权限/主题）放 Store
   - 页面级状态（列表数据/加载状态）放 Store 或 Hook
   - 组件级状态（弹窗开关/表单值）放组件内部
   - 明确状态变更链路：用户交互 → action → API → state 变更 → 视图更新
3. **API 设计**：
   - 先定义请求/响应的 TypeScript Interface
   - API 函数按业务域分组（`api/order.ts`、`api/user.ts`）
   - 复杂请求参数用 Interface 封装，不用过多散落的参数
4. **路由设计**：
   - 新增页面必须配置路由和懒加载
   - 需要权限的页面必须加路由守卫标记

#### 实现阶段

1. **编码顺序**：TypeScript 类型定义 → API 函数 → Store/Hook → 组件 → 页面
2. **组件编码约束**：
   - 单文件不超过 300 行（Vue SFC 不超过 500 行），超出必须拆分
   - 组件 props 必须定义类型和默认值，禁止用 `any`
   - 事件处理函数命名：`handle{Event}`，如 `handleSubmit`、`handleClick`
   - 副作用（定时器/事件监听）必须在 `onUnmounted`/`useEffect cleanup` 中清理
3. **测试规范**：
   - 工具函数（utils/hooks）必须有单元测试
   - 测试文件命名：`{filename}.test.ts` 或 `{filename}.spec.ts`
   - 组件测试聚焦交互逻辑，不测样式细节
4. **提交规范**：
   - 单次提交只包含一个功能点或一个 bugfix
   - 提交信息格式：`{type}: {description}`，type 为 feat/fix/refactor/style/test/docs
   - 样式调整独立提交，不与逻辑变更混合

## 知识库

知识库入口：`{知识库根目录}/知识库索引.md`（由 knowledge-generator skill 自动探测路径）
知识库根目录: `{知识库根目录}`（`$KB_ROOT` 变量值）

### 知识库消费规则

> 执行任何任务前，必须先查询知识库索引。

**三阶段消费+回退流程**：

1. **索引查询**（≤ 5秒）：查询知识库索引，匹配关键词
2. **命中质量评估**（≤ 3秒）：
   - **充分**（有≥1条coreChain + ≥1个入口方法）：消费知识库，按grepHints定位代码
   - **部分**（有coreClasses但无coreChain）：消费方向信息，补充Grep代码
   - **不足**（只有businessTerms或状态枚举碎片）：立即回退到全量Grep代码
   - **未命中**（索引无匹配）：直接全量Grep代码
3. **回退执行**（质量为"不足"或"未命中"时）：
   - 声明"知识库覆盖不足，回退到代码搜索"
   - 记录未覆盖场景到 `{知识库根目录}/.cache/missed-queries.log`
   - 直接执行 Grep/SearchCodebase 定位代码
   - 不在知识库中反复搜索浪费时间

> 未命中场景累计 3 次以上时，AI 应主动建议补充该子场景的知识库文档。

### 知识库维护规则

> 当任务涉及代码修改，且修改内容覆盖知识库已记录的模块、类、业务流程时，必须主动检查并同步更新对应知识库文档。

维护流程：
1. 代码修改完成后，检查本次修改涉及的类/模块
2. 在知识库索引中搜索是否有对应的知识库文件
3. 若有命中，读取对应知识库文件，更新受影响的内容
4. 更新完成后执行质量校验

### 禁止事项

- 禁止跳过知识库查询直接修改代码（除非知识库未命中且已回退到 Grep）
- 禁止修改代码后不更新知识库（当代码涉及已记录的业务逻辑时）
- 禁止删除知识库文件（只能标记为"已废弃"并说明原因）
